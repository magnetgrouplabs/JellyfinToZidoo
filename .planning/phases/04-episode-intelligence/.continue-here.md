Fix: Added guard `if (handlingPlaybackResult) { return; }` before the normal Zidoo result handler. Reset to false when new Zidoo player actually launches.
  - **Bug 2 FIXED: onStart() re-initialization** — When Zidoo player finishes and Play activity resumes, `onStart()` re-runs the FULL initialization including auto-play (`playButton.callOnClick()` at line 263 when debug mode is off). This caused the episode to replay from scratch. Fix: Added guard at top of `onStart()`: `if (handlingPlaybackResult || waitingForUpNext) { return; }`.
  - **Bug 3 FIXED: directPath overwritten by onStart()** — `onStart()` sets `directPath = originalIntent.getDataString()` (the HTTP stream URL). The Play Now handler's `doSubstitution` correctly sets `directPath` to the SMB path, but the async `getItem` callback references the instance variable `directPath` which `onStart()` could overwrite. Fix: Captured `final String resolvedSmbPath = directPath` immediately after `doSubstitution` and used that in the async callbacks instead of the instance variable.
  - **QoL added: finishWithResult()** — When finishing after a binge session (e.g. watched episodes 3, 4, 5), the Jellyfin client returns to episode 3 (the original intent). <decisions_made>

- **onStart() guard pattern**: `handlingPlaybackResult || waitingForUpNext` prevents re-initialization during Up Next flow. This is the key architectural fix — onStart() in the original PlexToZidoo was designed for single-play sessions and breaks when we chain multiple episodes.
- **Local variable capture for async safety**: `final String resolvedSmbPath = directPath` captures the SMB path before any async callbacks, preventing instance variable races.
- **handlingPlaybackResult as the central guard**: This flag now serves triple duty — prevents stale onActivityResult processing, prevents onStart re-init, and prevents onRestart->finish(). Reset only when new Zidoo player actually launches on UI thread.
- **finishWithResult() best-effort**: Sets result intent data with current jellyfinItemId. No guarantee Jellyfin client reads it, but zero risk.
- **USER FEEDBACK**: User proposed using Zidoo intents/API to advance episodes instead of Jellyfin path resolution. This was discussed but shelved because: (a) stopping the player means it can't receive "next" commands, (b) manual stop at <10% also needs Up Next but player is dead, (c) would require two code paths. Current intent-based approach handles both cases uniformly.
</decisions_made>

## Key Architecture Notes

- `onStart()` runs FULL initialization including auto-play when debug=off. Guards are essential when chaining episodes.
- `onActivityResult()` fires BEFORE `onStart()` in Android lifecycle — this is why the handlingPlaybackResult guard works.
- `finishActivity(98)` can generate stale/duplicate onActivityResult calls — the handlingPlaybackResult guard catches these.
- `showDebugPageOrSendIntent()` calls `playButton.callOnClick()` when debug is off (line 263) — this is why onStart re-init causes auto-play.
